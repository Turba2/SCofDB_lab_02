## Раздел 1: Описание проблемы

### Что такое race condition?

Race condition в базе данных — это ситуация, когда итог операции зависит от порядка выполнения нескольких конкурентных транзакций. Каждая транзакция отдельно выглядит корректной, но при одновременном запуске они читают устаревшее состояние и принимают решение на основе уже неактуальных данных.

В контексте маркетплейса это особенно опасно для оплаты заказа. Две параллельные сессии могут почти одновременно прочитать статус `created`, обе решить, что оплату можно проводить, и затем обе попытаться записать результат оплаты.

### Почему READ COMMITTED не защищает от двойной оплаты?

`READ COMMITTED` гарантирует только то, что каждая команда видит уже закоммиченные данные. Но это не означает, что вся транзакция работает с единым согласованным снимком данных.

В сценарии оплаты происходит следующее:

1. Первая транзакция читает статус заказа и видит `created`.
2. Вторая транзакция почти одновременно делает то же самое и тоже видит `created`.
3. Обе транзакции принимают одинаковое бизнес-решение: “заказ ещё не оплачен”.
4. После этого одна транзакция успевает обновить строку первой.
5. Вторая транзакция продолжает работу уже с решением, принятым на старом состоянии.

Если в приложении не проверять результат `UPDATE` и не использовать блокировки, можно получить две записи `paid` в `order_status_history`, хотя логически оплата должна быть только одна.

### Пошаговая демонстрация проблемы

| Время | Сессия 1 | Сессия 2 |
|------|------|------|
| t1 | `BEGIN` | |
| t2 | `SELECT status -> created` | |
| t3 | | `BEGIN` |
| t4 | | `SELECT status -> created` |
| t5 | искусственная задержка | искусственная задержка |
| t6 | `UPDATE orders SET status='paid'` | |
| t7 | `INSERT INTO order_status_history ... 'paid'` | |
| t8 | `COMMIT` | |
| t9 | | `UPDATE ... WHERE status='created'` возвращает 0 строк |
| t10 | | `INSERT INTO order_status_history ... 'paid'` |
| t11 | | `COMMIT` |

Итог: статус заказа становится `paid`, но в истории оказывается две записи об оплате. Это и есть прикладной race condition.

### Примеры из реальной жизни

1. Пользователь дважды нажал кнопку “Оплатить”.
2. Платёжный webhook пришёл повторно после сетевого сбоя.
3. Два фоновых воркера одновременно обрабатывают один и тот же платёжный event.
4. Мобильное приложение сделало retry после таймаута, хотя первая попытка ещё выполнялась.

## Раздел 2: Уровни изоляции в PostgreSQL

### READ UNCOMMITTED

`READ UNCOMMITTED` в классическом SQL допускает чтение незакоммиченных данных, но в PostgreSQL этот уровень фактически работает как `READ COMMITTED` из-за MVCC.

**Предотвращает:**
- грязные чтения в реальной реализации PostgreSQL

**Не предотвращает:**
- non-repeatable read
- phantom read
- прикладные race condition

**Когда использовать:**
- в PostgreSQL отдельно не имеет практического смысла, потому что ведёт себя как `READ COMMITTED`

### READ COMMITTED

Это уровень по умолчанию в PostgreSQL. Каждая SQL-команда видит только те изменения, которые были закоммичены к началу именно этой команды.

**Предотвращает:**
- dirty read

**Не предотвращает:**
- non-repeatable read
- phantom read
- потерю инварианта между “прочитал и решил” и “обновил”

**Когда использовать:**
- обычные CRUD-операции
- чтение каталога, списка заказов, истории
- большинство некритичных бизнес-сценариев

### REPEATABLE READ

`REPEATABLE READ` даёт транзакции стабильный snapshot. Все чтения внутри транзакции опираются на один и тот же снимок данных.

**Предотвращает:**
- dirty read
- non-repeatable read
- phantom read в PostgreSQL

**Не предотвращает полностью:**
- все сериализационные аномалии
- ошибки из-за неправильной логики обновления без явных блокировок

**Когда использовать:**
- для критичных операций, где важно консистентное чтение
- для оплаты, резервирования товара, изменения баланса

### SERIALIZABLE

Самый строгий уровень изоляции. PostgreSQL пытается обеспечить поведение, как будто транзакции выполнялись строго последовательно.

**Предотвращает:**
- все аномалии чтения
- write skew
- сериализационные конфликты на логическом уровне

**Недостатки:**
- больше rollback при нагрузке
- нужна retry-логика
- выше накладные расходы

**Когда использовать:**
- для коротких, очень критичных финансовых операций
- когда цена ошибки выше, чем цена повторного запуска транзакции

### Сравнение уровней

| Уровень | Dirty Read | Non-Repeatable Read | Phantom Read | Нагрузка | Подходящий сценарий |
|---|---|---|---|---|---|
| READ UNCOMMITTED | нет в PostgreSQL | да | да | низкая | отдельного смысла нет |
| READ COMMITTED | нет | да | да | низкая | обычные операции |
| REPEATABLE READ | нет | нет | нет в PostgreSQL | средняя | критичные бизнес-транзакции |
| SERIALIZABLE | нет | нет | нет | высокая | самые строгие сценарии |

## Раздел 3: Решение проблемы

### Почему REPEATABLE READ + FOR UPDATE решает проблему?

Решение состоит из двух частей:

1. `REPEATABLE READ` делает транзакцию предсказуемой и защищает от чтения “скачущего” состояния.
2. `FOR UPDATE` блокирует конкретную строку заказа, чтобы вторая транзакция не смогла одновременно принять решение по тому же заказу.

Практически это выглядит так:

1. Первая транзакция начинает оплату.
2. Она выполняет `SELECT ... FOR UPDATE` и получает строковую блокировку заказа.
3. Вторая транзакция пытается сделать тот же `SELECT ... FOR UPDATE`, но вынуждена ждать.
4. После commit первой транзакции вторая либо видит, что заказ уже оплачен, либо получает conflict/serialization error.
5. В приложении этот конфликт преобразуется в `OrderAlreadyPaidError`.

### Зачем нужен FOR UPDATE?

`FOR UPDATE` нужен потому, что один только `REPEATABLE READ` не даёт взаимного исключения. Он стабилизирует snapshot, но не запрещает конкурентным транзакциям одновременно пройти проверку бизнес-условия.

`FOR UPDATE`:
- ставит эксклюзивную блокировку на строку заказа
- заставляет вторую транзакцию ждать
- переводит проблему из “оба одновременно решили, что можно платить” в “только один реально работает с заказом”

### Что будет на REPEATABLE READ без FOR UPDATE?

Если использовать только `REPEATABLE READ`, обе транзакции всё равно могут начать работу, опираясь на снимок, где заказ ещё `created`. Дальше одна из них победит, а вторая:

- либо попытается обновить уже изменённую строку
- либо получит serialization/conflict error
- либо оставит прикладную логику в неудобном и труднообъяснимом состоянии

То есть `REPEATABLE READ` без `FOR UPDATE` улучшает консистентность чтения, но не даёт надёжного механизма “только один платёжный обработчик за раз”.

### Разница между FOR UPDATE и FOR SHARE

| Характеристика | FOR UPDATE | FOR SHARE |
|---|---|---|
| Тип блокировки | эксклюзивная | разделяемая |
| Блокирует UPDATE | да | да |
| Блокирует DELETE | да | да |
| Позволяет нескольким читателям брать одинаковую блокировку | нет | да |
| Основной сценарий | подготовка к изменению строки | безопасное чтение без изменения |

Для оплаты нужен именно `FOR UPDATE`, потому что после чтения мы сразу меняем статус заказа.

## Раздел 4: Рекомендации для продакшена

### Какой isolation level использовать для продакшена маркетплейса?

Оптимальная рекомендация для маркетплейса — использовать **READ COMMITTED как уровень по умолчанию**, а для критичных операций применять **REPEATABLE READ + FOR UPDATE** точечно.

### Почему не SERIALIZABLE везде?

Потому что это слишком дорого для повседневной нагрузки:

- увеличивается число конфликтов и rollback
- требуется retry-логика почти во всех критичных сценариях
- сложнее анализировать инциденты под нагрузкой

### Почему не READ COMMITTED везде?

Потому что в критичных сценариях этого недостаточно. Для оплаты, списания бонусов, резервирования остатков и изменения баланса нужен явный механизм сериализации доступа к конкретному ресурсу.

### Практический продакшен-подход

1. `READ COMMITTED` оставить дефолтным уровнем для API.
2. Для оплаты заказа использовать отдельный сервис с:
   - короткой транзакцией
   - `SET TRANSACTION ISOLATION LEVEL REPEATABLE READ`
   - `SELECT ... FOR UPDATE`
3. Обрабатывать конфликт как ожидаемую бизнес-ошибку, а не как “неизвестную проблему БД”.
4. Блокировать ресурсы всегда в одном порядке, чтобы снижать риск deadlock.

### Альтернативы

**SERIALIZABLE**
- даёт максимальную корректность
- требует retry-логики
- подходит не всегда из-за стоимости

**Optimistic locking**
- хороший вариант для распределённых систем
- работает через `version`/CAS
- удобен, если конфликты редки

**Advisory locks**
- полезны для логических ресурсов
- требуют больше дисциплины и аккуратности
- хороши как дополнительный инструмент, но не обязательны здесь

### Итоговая рекомендация

Для маркетплейса лучше всего подходит гибридный подход:

- `READ COMMITTED` для большинства запросов
- `REPEATABLE READ + FOR UPDATE` для критичных изменений состояния
- `SERIALIZABLE` только там, где цена ошибки максимальна и приемлем retry

## Заключение

В ходе лабораторной работы был построен сценарий конкурентной оплаты одного заказа и показано, что сама по себе корректная схема БД не решает все проблемы конкурентного доступа. Если приложение не использует блокировки и не учитывает реальный результат `UPDATE`, можно нарушить бизнес-инвариант даже при наличии ограничений на уровне таблиц и триггеров.

Основные выводы:

1. Race condition возникает не только из-за “плохой БД”, но и из-за неправильной последовательности действий в приложении.
2. `READ COMMITTED` недостаточен для операций, где важно атомарно выполнить “проверил состояние -> изменил состояние”.
3. `REPEATABLE READ + FOR UPDATE` даёт понятную и практичную защиту для конкурентных оплат.
4. Для продакшена лучше использовать не один “самый строгий” уровень изоляции везде, а выбирать стратегию под критичность операции.
